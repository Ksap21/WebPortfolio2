Homelab Infrastructure Project

Overview

This homelab is a self-hosted infrastructure environment designed around a VPN-first security model using Tailscale. All services are containerized using Docker and run on a single Ubuntu server. The goal is to provide secure remote access, service isolation, and practical experience with modern DevOps and networking tools. The entire system follows a zero public exposure approach, meaning no services are directly exposed to the internet.

Infrastructure

  • Operating System: Ubuntu Server
  • Local IP: 192.168.1.71
  • Remote Access IP: Tailscale (100.73.1.42)
  • Container Platform: Docker

Network Architecture

The homelab uses a VPN-first access model powered by Tailscale. All remote connections follow a secure encrypted tunnel into the homelab server, which then routes traffic internally to Docker services. 

Client Device → Tailscale VPN → Homelab Server → Docker Services

Containerized Services

Infrastructure Services

  • Pi-hole – Network-wide DNS filtering and ad blocking
  • Tailscale – Secure VPN layer for remote access
  • CasaOS – Lightweight Docker management interface

Application Services

  • Nextcloud – Self-hosted file storage and synchronization
  • Home Assistant – Smart home automation and device control
  • Ollama – Local large language model inference server
  • Open WebUI – Web interface for interacting with Ollama
  • Maybe Finance Stack – Personal finance system (app, worker, PostgreSQL, Redis)

Network Services & DNS

Pi-hole is deployed as a network-level DNS filter. It intercepts DNS requests from connected devices and blocks unwanted traffic such as ads and trackers before forwarding queries to external DNS providers. This improves privacy, performance, and control over domain resolution.

Traffic Flow Design

VPN Access Flow

Client Devices → Tailscale VPN → Server → Docker Services

DNS Resolution Flow

Devices → Pi-hole → External DNS Servers

Internal Service Communication

Docker containers communicate internally using bridge networks and service names, allowing secure interaction between services such as databases, APIs, and caches without external exposure.

Security Design Principles

  • No public port forwarding enabled
  • No direct internet exposure of services
  • All remote access secured via VPN (Tailscale)
  • Services isolated using Docker containers
  • Minimal attack surface design

Purpose of This Setup

  • Learning DevOps and networking concepts
  • Practicing secure infrastructure design
  • Running self-hosted applications
  • Experimenting with container orchestration
  • Building a cybersecurity-focused portfolio project

Summary

This homelab demonstrates a modern self-hosted architecture built around VPN-first secure access, containerized microservices, network-level DNS control, and a fully isolated application stack.
